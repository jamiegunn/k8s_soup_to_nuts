Every page in this guide that touches the wire mentions a timeout. [ingress-nginx](/networking/ingress-nginx/) has its 60-second read timeout, [Long-Lived Connections](/networking/long-lived-connections/) has the idle-timeout-kills-quiet-sockets pattern, [PostgreSQL](/stateful/postgresql/) has `statement_timeout`, [Graceful Shutdown](/workloads/graceful-shutdown/) has the grace period. Each is correct in isolation. Nobody owns the *sum* ‚Äî and the sum is where the outages live. A request crossing your corporate chain (client ‚Üí F5/NetScaler VIP ‚Üí MetalLB ‚Üí ingress-nginx ‚Üí Service ‚Üí pod ‚Üí Postgres) passes through **six independently-configured timers owned by four different teams**, and most "mysterious 504" and "random disconnect" tickets are two adjacent timers disagreeing by a few seconds.

This page is the ledger. It assumes you know each hop individually ‚Äî [Life of a Request](/routing/life-of-a-request/) for the path, [Front Door](/architectures/front-door/) for the topology ‚Äî and gives you the arithmetic that ties them together.

## Two timers, two rules

Every hop on the path runs (up to) two clocks, and they answer different questions:

| Timer | The question it answers | Fires when | Typical symptom when wrong |
|---|---|---|---|
| **Request timeout** | "How long will I wait for an *answer* to a request in flight?" | A response (or the next byte of one) doesn't arrive in time | 504 / `TimeoutException` at a suspiciously round number |
| **Idle timeout** | "How long will I keep a connection that's *saying nothing*?" | No bytes cross a connection in either direction for N seconds | Connection reset at exactly N seconds of quiet; "connection reset by peer" on the first use after a lull |

They need **opposite** orderings along the path, and both orderings have to hold at once:

**Rule 1 ‚Äî request timeouts shrink inward.** Each layer should wait slightly *less* than the layer in front of it: client 10s ‚Üí nginx 8s ‚Üí app handler 7s ‚Üí database 5s. Then the innermost, best-informed layer fails first ‚Äî your app returns a structured "database timed out" error, nginx relays it, the client gets an actionable 500-with-a-body instead of a bare 504 from a proxy that knows nothing. Honesty requires naming the other school: some shops run **edge-shortest** (the edge times out first and inner layers get generous timeouts), on the theory that the edge is the SLO enforcement point and inner work should be allowed to finish and warm caches even if the client already left. It's defensible ‚Äî but the inner request keeps consuming a database connection and a worker thread after nobody wants the answer, and the client's error is always the proxy's contentless 504. **Edge-longest (shrink inward) is this guide's default**: the caller times out *last*, so every error the client sees was generated by something that knew what went wrong.

The two schools side by side, since you'll meet both in design reviews:

| | Edge-longest (shrink inward) ‚Äî **default here** | Edge-shortest |
|---|---|---|
| Who fails first | Innermost layer (database, app handler) | The edge proxy / the client |
| Error the client sees | The app's structured error body, relayed outward | A bare 504 from a proxy that knows nothing |
| Server-side waste | None ‚Äî inner work is cancelled before anyone stops listening | Inner request runs to completion for an answer nobody reads, holding a DB connection and a worker thread |
| What it's good at | Diagnosable failures, bounded resource use | Hard SLO enforcement at one point; letting caches warm from abandoned work |
| Failure mode when misapplied | None structural ‚Äî just needs discipline at every hop | Retry storms: the client re-asks while the original still runs (see the multiplication below) |

**Rule 2 ‚Äî idle timeouts grow inward.** Each *server* should keep an idle connection slightly *longer* than whatever proxies to it, or the shorter timer closes sockets the longer side still considers live ‚Äî the sporadic-502 stale-keepalive classic from [Long-Lived Connections](/networking/long-lived-connections/): app idle keepalive **>** nginx upstream keepalive (60s) **>** ‚Ä¶ and where you can't out-wait a hop you don't own (the appliance's 300s), keepalives/pings shorter than the shortest idle timer on the path do the job instead.

:::caution[The rules point in opposite directions ‚Äî that's the whole trick]
Request timeouts: **smallest at the center**. Idle timeouts: **largest at the center**. Teams that learn Rule 1 often "fix" an idle disconnect by shrinking the app's keep-alive timeout to match ‚Äî and manufacture the stale-keepalive 502 documented in [front-door 5xx](/troubleshooting/front-door-5xx/). Check which timer fired *before* choosing which rule applies: request timeouts fire mid-request at a round number; idle timeouts fire between requests, at exactly N seconds after the last byte.
:::

## The hop table

Every timer on the corporate path, front to back. This is the page's backbone ‚Äî the worked budget below is this table with numbers filled in.

| Hop | Request-timeout knob | Default | Idle-timeout knob | Default | Who owns it |
|---|---|---|---|---|---|
| **Browser / JS client** | `fetch` + `AbortSignal.timeout()`; XHR `timeout` | none ‚Äî waits forever (until browser/OS gives up, minutes) | browser connection pool, not yours to tune | ~60‚Äì115s | Frontend team |
| **Java caller** (service-to-service) | `RestClient`/`RestTemplate` connect + read timeout; `WebClient` `responseTimeout` | **none** ‚Äî infinite unless set | pool `maxIdleTime` / `evictInBackground` | client-dependent | App team ‚Äî [Spring Boot on K8s](/java/spring-boot/) |
| **.NET caller** | `HttpClient.Timeout` | **100 seconds** | `PooledConnectionIdleTimeout` | 1 min | App team ‚Äî [ASP.NET Core on K8s](/dotnet/aspnetcore-on-k8s/) |
| **F5 / NetScaler VIP** | usually none at L4 (pass-through); L7 profiles vary | ‚Äî | TCP idle timeout | **300s** is the classic | Network team ‚Äî a ticket, not a YAML change ([External Load Balancing](/networking/external-load-balancing/), [Working with the Platform Team](/operations/working-with-platform-team/)) |
| **MetalLB** | none ‚Äî it's ARP/BGP address advertisement, not a proxy | ‚Äî | none | ‚Äî | Platform team |
| **kube-proxy / conntrack** | none ‚Äî DNAT has no opinion on HTTP | ‚Äî | `nf_conntrack_tcp_timeout_established` | 5 days (rarely the problem; table *capacity* is ‚Äî [NAT](/routing/nat/)) | Platform team |
| **ingress-nginx** | `proxy-connect-timeout` / `proxy-send-timeout` / `proxy-read-timeout` | **5s / 60s / 60s** | `upstream-keepalive-timeout` (to pods); `keepalive-timeout` (to clients) | 60s / 75s | Platform owns the ConfigMap defaults; **you own per-Ingress annotations** ([ingress-nginx](/networking/ingress-nginx/)) |
| **Tomcat (Spring Boot)** | `spring.mvc.async.request-timeout` (async only; sync requests have **no server-side cap**) | 30s async / ‚àû sync | `server.tomcat.keep-alive-timeout` | = `connection-timeout`, 60s stock | App team |
| **Kestrel (ASP.NET Core)** | none built-in; .NET 8+ request-timeouts middleware | ‚àû | `KestrelServerLimits.KeepAliveTimeout` | 130s | App team |
| **Hikari ‚Üí JDBC ‚Üí Postgres** | Hikari `connectionTimeout` (pool acquire); JDBC `socketTimeout`; Postgres `statement_timeout` | 30s / **0 = ‚àû** / **0 = ‚àû** | Hikari `idleTimeout`, `maxLifetime` | 10 min / 30 min | App team + DBA ([PostgreSQL](/stateful/postgresql/)) |
| **Valkey client** | Lettuce command timeout; Jedis `socketTimeout` | 60s / 2s | pool idle eviction + TCP keepalive | client-dependent | App team |
| **Partner API** | your HTTP client's timeout *for that call* | inherits the caller defaults above (100s!) | their edge's idle timeout ‚Äî ask | unknown until asked | You ask; they own |

Three defaults in that table cause a disproportionate share of incidents, so read them again: the .NET `HttpClient` **100-second** default (an eternity that silently violates every downstream budget), the JDBC `socketTimeout` **infinity** default (a hung database connection parks a thread forever), and the ingress **60-second** read timeout (the origin of nearly every "504 at exactly one minute" ticket in [front-door 5xx](/troubleshooting/front-door-5xx/)).

:::note[MetalLB and kube-proxy are the hops with no L7 opinion]
Nothing between the appliance and nginx parses HTTP, so nothing there can time out a *request* ‚Äî MetalLB only advertises the address, and kube-proxy's DNAT is per-packet with conntrack state that outlives any sane request ([NAT](/routing/nat/)). When a timeout fires on this path, the suspects are the appliance, nginx, the app, or the database. The kernel hops matter for *capacity* (conntrack table full drops SYNs), not for budgets.
:::

## The worked budget: orders-api in 10 seconds

Target: **the client sees an answer or a clean error within 10 seconds.** Decompose from the outside in, giving each layer slack below the layer in front of it ‚Äî enough to cover response serialization, one network round trip, and clock skew between machines that don't share a clock. 10‚Äì20% per hop is the working convention; below ~500ms of absolute slack you're betting against GC pauses.

| Hop | Knob | Value | Why |
|---|---|---|---|
| Client (Java `RestClient` / .NET `HttpClient.Timeout`) | read/total timeout | **10s** | The SLO itself. Times out *last* (Rule 1) ‚Äî everything inside fails first with a real error. |
| F5/NetScaler | ‚Äî (no request cap at L4) | idle 300s | Irrelevant to a 10s request; it's a Rule-2 citizen only. Confirm via ticket, don't assume ([Working with the Platform Team](/operations/working-with-platform-team/)). |
| ingress-nginx | `proxy-read-timeout` annotation | **8s** | 2s under the client: covers TLS, appliance transit, and nginx's own response write. Also `proxy-connect-timeout: 3` ‚Äî connecting to a pod should never take 5s. |
| App handler | `spring.mvc.async.request-timeout` / .NET request-timeouts middleware | **7s** | 1s under nginx, so *the app's* error body wins the race to the client. |
| Hikari pool acquire | `connectionTimeout` | **1s** | If no connection is free in 1s, the pool is exhausted and 29 more seconds (the default) won't fix it ‚Äî fail fast. |
| Postgres statement | `statement_timeout` (per-role or JDBC `options`) | **5s** | The real work. Sized from the query's p99 + margin, not from what's left over ‚Äî if p99 doesn't fit, the endpoint needs a redesign, not a bigger timeout. |
| JDBC socket | `socketTimeout` | **8s** | The backstop *above* `statement_timeout`: catches a dead server that can't send the timeout error. Never leave it at 0. |
| Valkey lookup | command timeout | **200ms** | A cache that's slower than that is a miss. Fall through to Postgres. |

Now audit the arithmetic bottom-up ‚Äî the budget must close:

```text
worst-case handler time = pool acquire + statement + cache + serialization
                        = 1s + 5s + 0.2s + ~0.3s  = 6.5s   ‚â§ 7s handler cap ‚úì
handler cap 7s + nginx overhead                  < 8s nginx read ‚úì
nginx 8s + edge transit                          < 10s client ‚úì
```

**The retry multiplication.** Retries spend the *same* budget again ‚Äî they don't get fresh time. One retry of a 5-second statement timeout is 10s of database wait inside a 7s handler cap: the retry itself causes the timeout it's retrying. The inequality every retry policy must satisfy:

```text
attempts √ó attempt_timeout + Œ£ backoff  ‚â§  the caller's own timeout
2 √ó 5s + 0.5s = 10.5s  >  7s   ‚úó  ‚Üí do not retry statement timeouts
2 √ó 3s + 0.5s =  6.5s  ‚â§  7s   ‚úì  ‚Üí 2 attempts fit only if each is capped at 3s
```

The practical rule: **retry fast failures (connection refused, pool-acquire timeout ‚Äî milliseconds to 1s), never slow ones (statement/read timeouts ‚Äî they already spent the budget).** This same multiplication is how a timing mismatch becomes a retry *storm* fleet-wide ‚Äî see the failure-signature table below.

And the budget as a timeline ‚Äî the worst-case request that still succeeds, and where each timer would have cut it off:

```text
t=0.0s   client sends            (client timer starts: fires at 10.0)
t=0.1s   nginx forwards to pod   (proxy-read-timeout starts: fires at 8.1)
t=0.2s   handler starts          (handler deadline: fires at 7.2)
t=0.4s   Valkey miss (‚â§0.2s)
t=1.4s   connection acquired     (Hikari would have failed at 1.4)
t=6.4s   statement returns       (statement_timeout would have fired at 6.4)
t=6.7s   handler serializes, responds       ‚Üê 0.5s under its own cap
t=6.9s   nginx relays                       ‚Üê 1.2s under proxy-read-timeout
t=7.0s   client has the answer              ‚Üê 3.0s under the SLO
```

Every gap between "would have fired" and "actually happened" is slack. If a step's slack is routinely near zero in your traces, that timer is the next incident ‚Äî widen the step's budget or shrink the step.

### Where each number lives

Budgets fail in the gap between the worksheet and the config files. The 10-second budget above, pinned in real syntax so nothing stays theoretical ‚Äî first the Java flavor:

```yaml
# application.yml ‚Äî orders-api (Spring Boot / Tomcat / Hikari)
spring:
  mvc:
    async:
      request-timeout: 7000          # handler cap ‚Äî async handlers only (see caution)
  datasource:
    url: jdbc:postgresql://orders-db:5432/orders?socketTimeout=8&options=-c%20statement_timeout=5000
    hikari:
      connection-timeout: 1000       # pool acquire: fail fast, 30s default is denial
      max-lifetime: 240000           # 4 min ‚Äî below the appliance's 300s idle (Rule 2)
      keepalive-time: 120000         # pings pooled connections through quiet periods
```

```java
// The outbound side ‚Äî this service calling the next one
RestClient restClient = RestClient.builder()
    .requestFactory(ClientHttpRequestFactorySettings.defaults()
        .withConnectTimeout(Duration.ofSeconds(2))
        .withReadTimeout(Duration.ofSeconds(10)))   // this caller's own 10s SLO
    .build();
```

:::caution[Tomcat has no request timeout for plain synchronous handlers]
`spring.mvc.async.request-timeout` only caps handlers that return `Callable`/`DeferredResult`/`WebAsyncTask`. A plain `@GetMapping` returning a value runs until *it* finishes ‚Äî the handler cap for sync code is the sum of its downstream timeouts (here 1s + 5s + 0.2s ‚âà 6.5s, inside 7 by construction). If an endpoint has no bounded downstream calls, it has no bound. That's why every row of the downstream section of the hop table must be finite ‚Äî they *are* the handler's timeout.
:::

The .NET flavor, where the handler cap is first-class middleware (.NET 8+):

```csharp
// Program.cs ‚Äî orders-api (Kestrel)
builder.Services.AddRequestTimeouts(o =>
    o.DefaultPolicy = new RequestTimeoutPolicy { Timeout = TimeSpan.FromSeconds(7) });
builder.WebHost.ConfigureKestrel(k =>
    k.Limits.KeepAliveTimeout = TimeSpan.FromSeconds(75));   // > nginx's 60s upstream keepalive

builder.Services.AddHttpClient("partner", c =>
    c.Timeout = TimeSpan.FromSeconds(10));                   // never ship the 100s default
app.UseRequestTimeouts();
```

And the edge, as annotations on the service's own Ingress (per-Ingress scope ‚Äî the fleet default stays untouched):

```yaml
metadata:
  annotations:
    nginx.ingress.kubernetes.io/proxy-connect-timeout: "3"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "8"      # seconds, no unit suffix
    nginx.ingress.kubernetes.io/proxy-send-timeout: "8"
```

The Postgres side deserves its own note: `statement_timeout` set per-role (`ALTER ROLE orders_app SET statement_timeout = '5s'`) beats setting it in the JDBC URL, because it also covers the ad-hoc psql session and the migration job that forgot ‚Äî and a per-role setting can't be silently dropped by one team's datasource refactor. Set the JDBC `socketTimeout` regardless: it's the only one of the pair that survives the database host dying mid-query ([PostgreSQL](/stateful/postgresql/)).

### The long-request exception lane

Reports, exports, bulk uploads ‚Äî real endpoints that legitimately need 300s. **Do not raise the fleet's `proxy-read-timeout` in the controller ConfigMap for them.** That converts every hung pod in the cluster from a 60-second 504 into a 300-second thread-and-connection hostage. The pattern is a *separate Ingress object* for the slow paths, carrying its own annotations:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: orders-api-reports        # separate object, same Service
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: "300"   # seconds, no unit
    nginx.ingress.kubernetes.io/proxy-send-timeout: "300"
    nginx.ingress.kubernetes.io/proxy-body-size: "100m"     # uploads
spec:
  rules:
  - host: orders.corp.example.com
    http:
      paths:
      - path: /reports
        pathType: Prefix
        backend: { service: { name: orders-api, port: { number: 80 } } }
```

Annotations scope per-Ingress, so `/reports` gets 300s while the rest of the host keeps the tight fleet default ([ingress-nginx](/networking/ingress-nginx/)). Two caveats: the *client* calling `/reports` needs its timeout raised too (Rule 1 still applies ‚Äî 300s nginx under a 100s `HttpClient` default just moves the failure), and anything beyond ~5 minutes should stop being a synchronous request at all ‚Äî return `202 Accepted` + a status URL.
## Failure signatures: when budgets disagree

Each row is a real mismatch between two adjacent hops, with its decoder. The timing of the failure *is* the diagnosis ‚Äî exact, repeatable numbers mean a timer, not a bug.

| Signature | What's actually happening | The mismatched pair | Decoder & fix |
|---|---|---|---|
| **504 at exactly 60.0s** ‚Äî `60.014s` in the access log, every time | Nobody set anything; nginx's default `proxy-read-timeout: 60` fired. The app is still processing; its answer arrives to a closed socket. | nginx (60s) vs app handler (‚àû) | `upstream_response_time` ‚âà 60 in the controller log is the smoking gun. Fix per Rule 1: app handler cap *below* 60, or annotate the Ingress if the endpoint is legitimately slow. Full walkthrough: [front-door 5xx](/troubleshooting/front-door-5xx/). |
| **Client gives up first ‚Üí load doubles** | Client timeout (say 5s) < server-side work (8s). The client abandons, retries, and now *two* copies of the request run to completion server-side. Under stress this compounds: each retry wave adds full-cost work for answers nobody reads ‚Äî the retry storm. | client (5s) vs everything inside (larger) | App logs show requests completing *after* the client disconnected; request rate spikes exactly at client-timeout intervals. Fix: restore Rule 1 (client waits longest), cap retries, and propagate cancellation (`CancellationToken` in .NET binds to the aborted request; check `Thread.isInterrupted`/servlet async timeout callbacks in Java). |
| **Reset at exactly N seconds of quiet** ‚Äî the 300s classic | An idle-timer murdered a keepalive/pooled connection mid-lull; the next borrow writes into a corpse ‚Üí `connection reset by peer`, sporadic 502s. The appliance's 300s TCP idle timeout is the usual culprit on this topology. | appliance idle (300s) vs pool `maxLifetime`/keepalives (longer or absent) | Failures cluster on the *first* request after quiet periods. Fix per Rule 2: pings/TCP keepalives < 300s, or pool `maxLifetime` below the shortest idle timer (Hikari `max-lifetime: 240000` vs a 300s proxy). Genre reference: [Long-Lived Connections](/networking/long-lived-connections/). |
| **5xx bursts timed exactly to deploys** | In-flight requests outlive a terminating pod. The drain window must fit inside **both** `terminationGracePeriodSeconds` *and* every hop's request timeout: `preStop sleep + longest request ‚â§ grace period`, and the longest request must itself fit the nginx/app caps or it was doomed regardless. | grace period (30s default) vs slowest endpoint (your `/reports` lane at 300s?) | If you built a long-request lane, its endpoints need a matching grace period on *their* pods ‚Äî or async redesign. Mechanics: [Graceful Shutdown](/workloads/graceful-shutdown/); the deploy-timeline math: [Rollout & Shutdown Knobs](/tuning/rollout-shutdown-knobs/). |

:::caution[The retry √ó timeout multiplication, one more time]
Layered retries multiply: client retries √ó2, its service-mesh or library retries √ó2, the app retries the database √ó2 ‚Äî one user click becomes up to **8 database statements**, each holding a connection for up to its full statement timeout. When the database slows down, this is the mechanism that turns "slow" into "down". Budget retries at **one layer** (usually the outermost caller), and only for fast failures.
:::

## Probes are timeouts too

The kubelet is another client with another request timeout: `timeoutSeconds` (default **1s**) is how long a single probe attempt may take before it counts as a failure. It belongs in the same budget thinking:

- The health endpoint must answer inside `timeoutSeconds` **while the app is at its worst** ‚Äî mid-GC, CPU-throttled, connection pool briefly exhausted. A probe that queries the database inherits the database's latency; three slow answers in a row and liveness restarts a pod that was merely busy.
- Rule 1 applies: whatever the probe handler *does* internally must be capped below `timeoutSeconds` ‚Äî a liveness endpoint with an uncapped dependency call is a restart generator.
- The probe budget interacts with the deploy budget: readiness `periodSeconds √ó failureThreshold` is your traffic-removal latency, which sits alongside the drain math above.

The dial-by-dial treatment (including why `timeoutSeconds: 1` is that page's biggest footgun) is in [Health Check Knobs](/tuning/health-check-knobs/).

## The audit kit

You can't budget what you can't see. Discovery, hop by hop:

**ingress-nginx ‚Äî what's set on your Ingresses, and what the fleet default is:**

```bash
# Per-Ingress overrides (empty output = you're on defaults)
kubectl get ingress -n orders -o json | jq -r '
  .items[] | .metadata.name as $n
  | .metadata.annotations // {} | to_entries[]
  | select(.key | test("timeout|keepalive"))
  | "\($n)\t\(.key)=\(.value)"'

# Fleet defaults: the controller ConfigMap, then the rendered truth
kubectl get cm -n ingress-nginx ingress-nginx-controller -o yaml | grep -i timeout
kubectl exec -n ingress-nginx deploy/ingress-nginx-controller -- \
  grep -E 'proxy_(connect|send|read)_timeout|keepalive_timeout' /etc/nginx/nginx.conf | sort -u
```

```text
orders-api-reports   nginx.ingress.kubernetes.io/proxy-read-timeout=300
orders-api-reports   nginx.ingress.kubernetes.io/proxy-send-timeout=300
                     ‚Üê orders-api itself printed nothing: it's on fleet defaults
proxy_connect_timeout   5s;
proxy_read_timeout      60s;      ‚Üê the rendered truth; annotations appear per-location
proxy_send_timeout      60s;
```

The rendered `nginx.conf` outranks everything: if the annotation you set isn't in a `location` block there, it isn't in effect (typo'd key, wrong ingress class ‚Äî annotations fail silently).

**Your app and its clients** ‚Äî grep config, not memory. Java: `connectTimeout|readTimeout|responseTimeout|connection-timeout|socketTimeout|statement_timeout|max-lifetime` across `application*.yml` and datasource URLs. .NET: `HttpClient.Timeout`, `KeepAliveTimeout`, request-timeouts middleware policies in `Program.cs`. Postgres side: `SHOW statement_timeout;` per role ‚Äî per-role settings silently override your JDBC options.

**The appliance** ‚Äî you can't read it; ask for it. The ticket template that gets a useful answer ([Working with the Platform Team](/operations/working-with-platform-team/)):

> For VIP `orders.corp.example.com:443` ‚Üí MetalLB pool `10.40.8.0/24`: (1) TCP idle timeout on the virtual server and its profile? (2) Any L7/HTTP profile with a request/response timeout, or pure L4 pass-through? (3) Does the VIP SNAT, and what's the SNAT idle timeout? (4) Health-monitor interval/timeout against the MetalLB IP?

**The worksheet.** One page per service, filled in from discovery above, kept in the repo next to the manifests. If a cell is "unknown", that's a finding:

| Hop | Request timeout | Idle timeout | Set where | Verified how | Rule check |
|---|---|---|---|---|---|
| Client(s) | ___ | ___ | ___ | config grep | largest request timeout on the path? |
| Appliance | ___ | ___ | ticket #___ | platform answer | idle > our ping interval? |
| ingress-nginx | ___ | ___ | annotation / ConfigMap | `nginx.conf` grep | < client, > app? |
| App server | ___ | ___ | ___ | config grep | < nginx? idle > nginx keepalive? |
| Pool + DB | ___ | ___ | ___ | `SHOW statement_timeout` | < handler? `maxLifetime` < shortest idle? |
| Cache / partner | ___ | ___ | ___ | ___ | fits inside handler? |
| Probes | `timeoutSeconds`: ___ | ‚Äî | pod spec | `kubectl get pod -o yaml` | > health-endpoint p99 under load? |
| Grace period | `terminationGracePeriodSeconds`: ___ | ‚Äî | pod spec | pod spec | > preStop + slowest request? |

**Change discipline.** Timeout changes look trivial and interact non-locally, so treat them like schema migrations: **one hop per change**, through git, with the worksheet updated in the same commit stating which rule (shrink-inward or grow-inward) the new value satisfies and against which neighbor. Verify with a canary request that *should* time out (a `pg_sleep()` endpoint in non-prod) and confirm the failure happens at the layer you intended, at the number you set. Then ‚Äî and only then ‚Äî the next hop.

## The checklist

- [ ] Every hop's two timers are *known* (worksheet has no "unknown" cells) ‚Äî including the appliance's, via ticket.
- [ ] Request timeouts shrink inward: client > nginx > handler > statement, with ‚â•500ms slack per step.
- [ ] Idle timeouts grow inward ‚Äî or keepalives/pings run shorter than the shortest idle timer you don't control.
- [ ] No infinities: JDBC `socketTimeout` set, .NET `HttpClient.Timeout` consciously chosen (not the 100s default), sync handlers capped.
- [ ] Retries satisfy `attempts √ó timeout + backoff ‚â§ caller's timeout`, retry fast failures only, at one layer.
- [ ] Slow endpoints ride the exception lane (separate Ingress + annotations), not a raised fleet default.
- [ ] Grace period ‚â• preStop + slowest request the pod can be holding.
- [ ] Probe `timeoutSeconds` > health endpoint p99 under load.
- [ ] Timeout changes ship one hop at a time, through git, with the worksheet in the diff.
